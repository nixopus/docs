</Info>

A machine is the dedicated compute environment where your apps run. Each machine has its own CPU, memory, storage, and networking — fully isolated from other users.

## Provisioning

When you sign up and create your first organization, Nixopus automatically provisions a machine for you. There's nothing to configure — the platform detects that you don't have a machine yet and starts the provisioning process.

### What happens behind the scenes

1. **Initializing** — Nixopus allocates resources for your machine.
2. **Creating machine** — The compute environment is created and configured.
3. **Configuring SSH** — Secure access is set up so you can use the built-in terminal.
4. **Setting up networking** — Port forwarding, reverse proxy, and your `*.nixopus.ai` subdomain are configured.
5. **Verifying** — Nixopus confirms the machine is reachable and healthy.
6. **Complete** — Your machine is ready. You'll see a notification: "Your machine is ready!"

Provisioning typically takes under a minute. If it fails, you can retry from the dashboard — Nixopus will pick up where it left off.

## Machine states

Your machine can be in one of these states:

| State | Description |
|-------|-------------|
| **Provisioning** | Machine is being set up for the first time. |
| **Running** | Machine is active and serving your apps. |
| **Paused** | Machine is suspended. Apps are not accessible. No compute is consumed. |
| **Stopped** | Machine is shut down. Data is preserved. |
| **Failed** | Provisioning or a lifecycle action failed. Retry from the dashboard. |

The current state is always visible on your machine card in the dashboard.

## Lifecycle actions

You can manage your machine from the actions menu on the machine card:

### Restart

Restarts the machine. Use this if your apps are behaving unexpectedly or after configuration changes that require a fresh start. Your apps will experience brief downtime during the restart.

<Warning>
  If your apps are crashing due to out-of-memory (OOM) errors, restart the machine to reclaim memory. If it keeps happening, your workload may need more resources — see [Upgrading](#upgrading).
</Warning>

### Pause

Suspends the machine. Your apps stop serving traffic and no compute is consumed while paused. Data and configuration are preserved. Use this when you're not actively using the machine and want to save credits.

### Resume

Brings a paused machine back to the **Running** state. Your apps become accessible again within seconds.

## Backups

Nixopus can back up your machine's data to S3-compatible storage using [restic](https://restic.net). Backups capture the contents of key directories and store them as deduplicated, encrypted snapshots.

### What gets backed up

By default, the following directories are included in every backup:

- `/home` — user home directories and app data
- `/etc` — system configuration
- `/var/lib/docker/volumes` — persistent Docker volume data

You can override the default paths with your own list in **Organization Settings → Backups**.

### Triggering a backup

To start a backup immediately, go to your machine card and select **Back up now** from the actions menu. You can also trigger one via the API:

```bash
curl -X POST 'https://api.nixopus.com/api/v1/machines/backup?server_id=<server-id>' \
  -H 'Authorization: Bearer <token>' \
  -H 'X-Organization-Id: <org-id>'
```

A backup job is queued and runs asynchronously. The response includes a `request_id` you can use to correlate the job.

### Viewing backup history

Past backups are listed under **Machine → Backups** in the dashboard, or via the API:

```bash
curl 'https://api.nixopus.com/api/v1/machines/backups?server_id=<server-id>' \
  -H 'Authorization: Bearer <token>' \
  -H 'X-Organization-Id: <org-id>'
```

Each entry shows the status (`pending`, `in_progress`, `completed`, `failed`), snapshot path, S3 path, and size.

### Scheduled backups

Enable automatic backups on a schedule from **Organization Settings → Backups**:

| Setting | Options | Default |
|---------|---------|---------|
| **Frequency** | `daily`, `weekly` | `daily` |
| **Hour (UTC)** | 0–23 | `2` |
| **Day of week** | 0 (Sun) – 6 (Sat) | `0` |
| **Retention** | Number of snapshots to keep | `7` |

The scheduler runs hourly and enqueues a backup for any org whose configured time matches the current UTC hour (and day, for weekly). A backup is skipped if one is already in progress or the last completed backup is too recent (within 20 hours for daily, 6 days for weekly).

### Backup storage

Backups are stored in your configured S3-compatible bucket under the path:

```
backups/<org-id>/<server-id>/snapshots/<snapshot-id>
```

The restic repository is encrypted. The encryption key is derived from your server ID and is never written to the remote machine.

<Note>
  Backup requires S3 storage to be configured. If S3 is not set up, triggering a backup returns an error. Contact support if you need help configuring storage.
</Note>

## Machine resources

Each machine comes with allocated resources based on your plan:

| Resource | Pro |
|----------|-----|
| **vCPU** | 2 |
| **Memory** | 2 GB |
| **Storage** | 25 GB |
| **Bandwidth** | 1 TB |

Resources are dedicated to your machine — not shared with other users.

### Upgrading

Need more CPU, memory, or storage? Contact us at [raghav@nixopus.com](mailto:raghav@nixopus.com) with your organization name and the resources you need. Self-service machine upgrades are on the roadmap.

## Adding machines

<Info>
  Multi-machine support is coming soon.
</Info>

Nixopus is designed to support multiple machines per organization. You'll be able to add machines in different regions, distribute apps across them, and manage your fleet from one dashboard.

Planned capabilities:

- **Add machines** — spin up additional machines from the dashboard
- **Replicate** — duplicate an existing machine's configuration to a new instance
- **Multi-region** — deploy machines across regions for lower latency
- **Load balancing** — distribute traffic across machines automatically

Want early access to multi-machine features? Reach out at [raghav@nixopus.com](mailto:raghav@nixopus.com).

## Machine billing

Machine usage is part of your Nixopus Cloud plan. Your subscription covers the compute resources allocated to your machine. See [credits](/cloud/credits) for how usage billing works.

When your machine is **paused**, no compute is consumed — only storage is retained.

<Note>
  For plan details and pricing, see [nixopus.com/pricing](https://nixopus.com/pricing).
</Note>
