## What makes Nixopus different

### Self-healing deployments

On other platforms, a failed deployment means context-switching: read the logs, copy the error to an AI tool, get a fix, push it, redeploy, wait. Repeat until it works.

Nixopus closes that loop. The agent detects the failure, reads the build or container logs, creates a pull request with the fix, and redeploys. You review the PR or let the agent handle it.

### Autonomous monitoring

Deploying on a Friday night shouldn't mean staying up to watch logs. The agent will monitor deployments after they go live — if a container crashes at 2am, it diagnoses the issue, raises a PR, and redeploys while you sleep.

### Open source and self-hostable

Fully open source — run on your own machine with the same features as Nixopus Cloud. Your code, your data, your infrastructure.

### Full machine access

[Built-in terminal](/guides/terminal) and [containers](/guides/containers) — direct access to your infrastructure when you need it.

### Multi-machine orchestration

Connect as many machines as you need — 2, 10, or 100 — and manage them all from one place. Multi-machine deployments, load balancing, and automated scaling are on the roadmap.

### No framework lock-in

Anything that runs in a container deploys — Next.js, Django, Rails, Go, FastAPI, or a multi-service stack with Docker Compose.

### No surprise bills

Credit-based pricing — you add credits, spending stops when your balance runs out. Self-hosted is free forever.
