Run the full Nixopus stack on a machine you control.

```bash
curl -fsSL install.nixopus.com | sudo bash
```

<Warning>
  Nixopus is in active development. Self-hosted images are pulled as `latest` and are not pinned to a specific version. Use in production workloads at your own risk.
</Warning>

## When to self-host

**Self-host** when you need full control over your data, have specific compliance requirements, or want to run on your own infrastructure.

**Use Nixopus Cloud** for production workloads. Nixopus Cloud includes security patches, pinned and tested image versions, automatic backups, and priority support. Sign up at [dashboard.nixopus.com](https://dashboard.nixopus.com) ŌĆö you get free allocated machine resources on signup to explore the platform.

## Requirements

Nixopus manages Docker, binds ports 80/443, and SSH-es into the host. **Use a fresh, dedicated VPS** ŌĆö not a machine already running other production services. Shared machines will have port conflicts, permission issues, and risk interfering with existing workloads.

| Requirement | Minimum |
| --- | --- |
| Machine | Fresh VPS from any cloud provider (Hetzner, DigitalOcean, AWS, etc.) |
| Architecture | x86_64 (amd64) or aarch64 (arm64) |
| RAM | 1 GB minimum (2 GB+ recommended) |
| Disk | 2 GB free minimum |
| Access | Root (the installer must run as root) |
| Docker | Installed automatically if not present (Docker Engine + Compose V2) |

### Tested distributions

Tested in CI on every release:

| Distribution | Version |
| --- | --- |
| Ubuntu | 22.04, 24.04 |
| Debian | 12 |
| Rocky Linux | 9 |
| Alpine | 3.20 |

### Should also work

The installer has support paths for these but they are not tested in CI:

| Distribution | Notes |
| --- | --- |
| Alma Linux | Uses the same install path as Rocky |
| CentOS / RHEL | Uses the same install path as Rocky |
| Fedora | Uses `dnf`, same as Rocky/Alma |

Other Linux distributions may work if Docker and Compose V2 are already installed. The installer requires `/etc/os-release` to be present.

## Install

<Steps>
  <Step title="Point DNS (optional)">
    If you want automatic HTTPS, create a DNS A record pointing your domain to the machine's public IP **before** running the installer.

    Skip this step to run in IP mode over plain HTTP.
  </Step>
  <Step title="Build your install command">
    Configure your installation using the builder below. Select your options and the command updates in real time ŌĆö then copy and run it on your server.

    <InstallCommandBuilder />

    See [Configuration](/self-hosting/configuration) for detailed documentation on every option.
  </Step>
  <Step title="Access the dashboard">
    Once services are healthy, open your browser and navigate to:

    - **With domain:** `https://your-domain.com`
    - **Without domain:** `http://your-machine-ip`

    If you provided `ADMIN_EMAIL`, the installer creates the admin account for you and prints the credentials in the install summary. If you also provided `ADMIN_PASSWORD`, that's used; otherwise a strong password is auto-generated and printed once. The same values are stored in `/opt/nixopus/.env` (mode `600`).

    If you skipped `ADMIN_EMAIL`, the dashboard sends you to `/register` on first visit ŌĆö the first user to sign up becomes the admin.
  </Step>
</Steps>

## Installer log

The installer saves everything printed to the terminal to **`/opt/nixopus/install.log`** (mode `600`) when the install directory exists. If the installer fails before that directory is created, the transcript stays under **`/tmp/nixopus-install-*.log`** ŌĆö the failure message prints the path.

The log can include passwords or API keys if they appear in the output. Review and redact before sharing.

For a single paste-friendly bundle **with secrets stripped from `.env`** (and recent compose logs), run:

```bash
sudo nixopus report
# optional: save to a file
sudo nixopus report > /tmp/nixopus-report.txt
```

See [Management CLI ŌĆö report](/self-hosting/management-cli#support-report) and [Troubleshooting](/self-hosting/troubleshooting#reporting-an-issue-install-or-runtime).

## What the installer modifies

Everything the installer touches outside of `$NIXOPUS_HOME` (`/opt/nixopus` by default):

| Change | Path | Notes |
| --- | --- | --- |
| Installs prereqs if missing | System packages | `curl`, `openssl`, `openssh-client` via apt/dnf/apk |
| Installs Docker if missing | System packages | Docker Engine + Compose V2, enabled on boot |
| Management CLI | `/usr/local/bin/nixopus` | Overwritten on each install |
| SSH public key | `~/.ssh/authorized_keys` | Appended once (skips if already present). Required for deployments via SSH. |

Everything else (config, compose files, SSH keys, Caddyfile) is contained in `$NIXOPUS_HOME`.

<Note>
  Whoever owns the first credential account becomes the admin ŌĆö either the user the installer pre-creates from `ADMIN_EMAIL`/`ADMIN_PASSWORD`, or the first person to register through the dashboard if you skipped those. After that, registration is closed ŌĆö you invite users manually.
</Note>

<Tip>
  If the auto-bootstrap fails (services slow to start, origin mismatch, etc.) you can retry without reinstalling: `sudo nixopus admin-bootstrap`. See [Management CLI](/self-hosting/management-cli#admin-bootstrap) and [Troubleshooting](/self-hosting/troubleshooting) for details.
</Tip>
